# 三福HD / P・SPO 本格運用 移行設計書

デモ（完全静的 no-build PWA・12HTML/22JS/5層CSS・バックエンド無し）から、3本柱（守り=会員UX / 攻め=データ×AI / 新規=B2Bプラットフォーム）を支える本格運用基盤へ。6レンズ（モバイル/Web面分割/バックエンド/AI-ML/リポジトリ-CI/セキュリティ）を統合し、矛盾を解消した単一の実行計画。**前提を一つだけ先に固定する**: 地方・小規模・学生BPO＋AI実装者という体制では、「自前インフラを建てない・マネージドに寄せる・no-build資産を捨てない・PIIと生体を最小の1箇所に閉じ込める」が全章を貫く設計憲法である。

**レンズ間整合の確認**:
- **モバイル⇔Web面分割**: Capacitorは「会員面（app/me/area/live＋price/guide閲覧）」だけをラップし、同じ`www/`をWeb版`app.pspo.jp`とも共有。経営面（growth/data）は**アプリに入れない**＝両レンズが「顧客と経営は別系統」で一致。
- **Web面分割⇔バックエンド**: ドメイン分割＝認証分割＝バンドル分割を一致させ、「顧客バンドルに`insights.js`が物理的に存在しない」状態を作る。これがBFFのPII境界（分析ゾーンを認証越しにのみ公開）と同じ線。
- **バックエンド⇔AI/ML⇔セキュリティ**: 3レンズが独立に「BigQuery+dbt / member_sk擬似ID / 顔特徴量非保有 / 月2万円以下 / マネージド優先」へ収束。クラウドはGCP+Supabaseに集約（マルチクラウドにしない）。
- **クラウド選定の1点だけの不整合と裁定**: レンズ5（リポ/CI）はホスティングにVercelを推奨、他レンズはCloudflare/GCP/Supabaseを推奨。**裁定**: ホスティングは**Cloudflare Pages/Workersに統一**する（理由は§5・§7で詳述。Cloudflare AccessがそのままZero-Trust社内認証になり、レンズ2/6の「ops.pspo.jpをAccessで保護」と一致し、Supabase/BigQueryとの結合も薄い）。Vercelの利点（Preview環境）はCloudflare PagesのPreview Deploymentsで代替でき、ベンダーロックを浅く保てる。

## 4. モバイル（iOS/Android）戦略の確定方針

### 確定方針: Capacitorラッパー一択（新規ストアアカウント・新Bundle ID）

デモが`<script type="module">`で`./js/*.js`を直importし、外部fetchがゼロ（serviceWorker登録とgeolocationのみ・コードで確認済）であることから、**Capacitorの`www/`にディレクトリごと置けばそのまま動く**。約7,185行のUI資産を無改修で再利用でき、追加実装はプッシュ/Wallet/オフライン/アプリ内Browserの薄いプラグイン層のみ。Swiftフルネイティブ（iOS/Android二重保守）・RN/Flutter（DOM直書きUIの全書き直し＝デモ資産が死ぬ）は、学生BPO＋AI実装者体制では1年Phase1を回せず非推奨。

### 確定事項

| 論点 | 確定 | 理由（地方小規模の現実） |
|---|---|---|
| 実装方式 | **Capacitor**（@capacitor/ios + android） | no-build素ES Modules資産を保全。Web層とアプリで同一`www/`を共有 |
| ストアID | **自社新規アカウント＋新Bundle ID `jp.pspo.app`** | 旧OEM（couponapp2系・id1499960318）はベンダー所有でApp Transfer交渉が重い。法人D-U-N-SでApple/Google開発者登録 |
| 旧アプリ | **段階退役**（移行バナー→説明文で告知→DL/MAU逆転後に公開停止） | 既存ユーザーのリーチ喪失を最小化 |
| 顔認証 | **アプリで照合しない**。来館タイムスタンプ結果だけ受領 | 生体をアプリ/スタックに持ち込まない＝APPI・ストア審査・非PII方針すべてに整合 |
| オフライン | **Capacitor同梱アセットを正**、sw.jsは補助 | WKWebView/Android WebViewでSW挙動が不安定。初回起動オフライン保証 |
| 更新 | **Web層OTA**（capacitor-updater等）＋ネイティブ追加時のみ審査 | insights.js等のデータ更新・バグ修正を当日反映。審査待ち停滞を回避 |
| ネイティブ機能 | プッシュ（FCM/APNs・p8キー）/ Wallet会員証（PassKit/Google Wallet・member.jsのRANKS対応）/ アプリ内Browser（hacomono導線の体験分断解消） | App Store審査ガイドライン4.2（純WebViewラッパー却下）対策＝「WebにないネイティブUX」を初回提出から同梱 |

### ストア提出の現実ブロッカー（コードから判明・必修）
1. **アイコンが全SVG**（manifest.webmanifest:15-17で確認）→ App Store/PlayはPNG必須。192/512/1024＋iOS各サイズ＋Androidアダプティブを生成。
2. **平文Mapsキー**（config.local.js: `AIzaSyAKiqK...`）→ Capacitorバンドルに同梱＝実質公開。**即ローテーション＋Bundle ID/署名制限のネイティブMapsキーに分離**、Web版とアプリ版で鍵を分ける。
3. **robots全Disallow＋デモ免責コピー**→ 本番文言へ。
4. **WebView主体の審査リスク**→ プッシュ/Wallet/オフライン/ネイティブ地図を同梱。

## 5. バックエンド / データ基盤（スキーマ例込み）

### 3層 + クリーンルーム境界（マネージド優先）

**(1) BFF/正本**: Supabase（Postgres + Auth + RLS + Edge Functions・東京リージョン）。hacomonoが契約/課金/予約の正本、顔認証ゲートが来館の正本、Supabaseは「非PII射影のミラー＋アプリ独自状態＋最小PII」に徹する。
**(2) イベント収集**: 顔認証/hacomono → Webhook/CSVバッチ → Edge Function `ingest` で受信 → 顔特徴量は受け取らず破棄、生member_idを`HMAC-SHA256(id, salt)`で擬似匿名化 → `visit_events`（Postgres）とObject Storage（Parquet）へ。
**(3) DWH/クリーンルーム**: BigQuery（オンデマンド課金＝夜間アイドルの長いジムに最適）に**非PIIのみ**日次ロード、dbtで`stg→int→mart`構築。B2B/社内はmartだけ参照しPII正本に到達不能。

**API様式**: REST + OpenAPI（GraphQL不採用＝小チームの保守性・DWHコスト予測性・Supabase自動RESTとの相性）。バージョン`/v1`。

### データモデル（PII分離を最初から構造化）

```sql
-- ① PIIゾーン（Supabase secure スキーマ・BigQuery複製禁止・社内最小権限RLS）
create table secure.members_pii (
  member_sk        text primary key,           -- HMAC-SHA256(hacomono_cust_id, salt)
  hacomono_cust_id text unique not null,        -- hacomonoの正本キー
  face_gate_ref    text,                        -- 顔認証ベンダの会員参照ID（特徴量ではない）
  name text, phone text, birth_date date, gender text,
  plan text, joined_at date, churned_at date,
  created_at timestamptz default now()
);  -- salt はDBに置かずSecret Manager。member_sk生成はEdge Functionのみ。

-- ② 分析ゾーン（app スキーマ → BigQueryへ非PII射影）
create table app.visit_events (              -- 来館イベント（顔認証の非PII射影・append-only）
  event_id uuid primary key default gen_random_uuid(),
  gate_event_id text unique,                 -- べき等キー（重複排除）
  member_sk text not null,                   -- 擬似IDのみ。生member_id/顔は持たない
  store_id text not null references app.stores(store_id),
  occurred_at timestamptz not null,
  direction text,                            -- in/out（在館人数=in−out集計の素材）
  age_band text, gender text,                -- 個票でなく帯
  ingested_at timestamptz default now()
);
create index on app.visit_events (store_id, occurred_at);

create table app.stores (                    -- stores-real.js + geo.js 統合
  store_id text primary key, name text not null, area text not null,
  category text, addr text,
  lat double precision, lng double precision,
  geo_precision text,                        -- 'exact'/'area'（概略明示）
  is_active boolean default true
);
create table app.zones (                     -- data.js zone/gear
  zone_id text primary key, store_id text references app.stores(store_id),
  kind text, label text, capacity int
);
create table app.occupancy_snapshot (        -- 占有時系列（頭数のみ・映像なし）
  store_id text references app.stores(store_id),
  zone_id text, observed_at timestamptz not null,
  occupied int, capacity int, ratio numeric,
  primary key (store_id, zone_id, observed_at)
);

-- ③ 同意台帳（consent.jsを追記式に正本化＝APPI/監査対応・履歴を消さない）
create table app.consent_ledger (
  consent_id uuid primary key default gen_random_uuid(),
  member_sk text not null,
  purpose text not null,                     -- 'service_improve'/'anon_aggregate'/'b2b_provide'
  granted boolean not null, source text,
  occurred_at timestamptz default now()
);  -- 「現在の同意」はpurposeごと最新行をビューで導出。撤回も追記で残す。
```

### 認証3系統
- **会員**: hacomono OIDC委譲が第一候補（自前パスワード非保持・退会反映が正本に従う）。未対応ならSupabase Auth phone（SMS OTP）で独自セッション＋hacomono顧客番号を日次CSV突合。`auth.js`のlocalStorage平文電話番号は本番で撤去。
- **社内**: Cloudflare Access + Google Workspace SSO（既存会社メール基盤・ゼロ追加コスト）＋IP制限。**これが現状最大リスク（growth/data素HTML公開）への直接の解**。
- **B2B**: Clerk等マネージドIdP（契約企業の招待制・将来のSSO拡張）。

### 段階移行の安全策（「動くデモ」を一度も壊さない）
フロントは**静的フォールバック温存**（`/v1/stores`が落ちてもstores-real.js等で描画継続＝オフラインの強みを壊さない）。store/occupancyの**読み取り**だけAPI化→consent書き込み→member派生→来館パイプライン、の順で1機能ずつ。

## 6. AI/ML導入順序

**原則: ゼロから作るのではなく、既存ロジック（stats.jsのプロトタイプ・insights.jsのAI_PLAYBOOK・CHURNの教師ラベル）を実データに接地する移行。** スタックは「Warehouse中心・バッチファースト・マネージドLLM」＝BigQuery+dbt+Cloud Run Jobs+Claude API。リアルタイム推論サーバ/Feast/k8s/GPU/有償ETLは避け、**月2万円以下**を維持。feature storeは専用製品でなくBigQuery上の`features_member_daily`/`features_store_hourly`を「学習と推論で同一SQL」で生成しスキューを防ぐ規約で代替。

| 順 | モデル | 手法 | なぜこの順か | 受け入れ基準 |
|---|---|---|---|---|
| ① 最優先 | **離反予測** | LightGBM（30日離反確率）＋lifelines（生存曲線） | 教師信号が既存（CHURN）・追加センサ不要・LTV直撃（退会防止LTV≒7.7万円>CAC） | churn AUC≥0.75、上位decileのlift、calibration、A/Bでリテンション差検証後に自動化 |
| ② | **占有/需要予測（店舗総量）** | Prophet or LightGBM | 顔認証は入退館のみ＝設備別占有データ無し。in−out差分で在館純人数のみ予測可。カメラレス維持 | 時間帯予報の誤差、オフピーク誘導クーポンの効果測定 |
| ③ | **クロスセル推薦** | implicit(ALS)/LightFM | 「多機能すぎて1施設しか使わない割高感」を直撃 | 推薦経由の新規業態利用率・利用業態数増 |
| ④ | **社内BI LLM**（会員サポートより先） | Claude（Haiku=ルーティング/要約、Sonnet=BI推論）＋RAG | 閉じた環境で幻覚許容・PII露出制御しやすい | PIIを渡さない（代理ID+集計のみ）プロンプト境界、月数千〜2万円 |
| ⑤ | **行動系列(自己教師あり)＋卒業予測＋会員サポートLLM** | 系列埋め込み/lifelines/限定RAG | 18歳離脱（最大課題）に卒業予測で先回り。サポートは料金/予約手順に限定 | feature注入で①③の性能底上げ、サポートの幻覚/費用抑制 |

**設備別占有の確定方針**: 当面は店舗総量のみ。検証後に**非PII優先で**更衣室/サウナ=サーマル（Butlr型・非カメラ）、ジムフロアのみ頭部数YOLO（顔非保持）を1-2店パイロット。全店カメラAIは初手で導入しない。

**MLOps（人手最小）**: Cloud Scheduler→Cloud Run Jobs（日次バッチ→predictionsテーブル書き戻し→アプリは読むだけ）。占有のみ軽量Cloud Run API。MLflow（レジストリ）＋Evidently（日次ドリフト）＋週次再学習cron＋しきい値割れSlack通知。**LLMにはPIIを渡さない**（代理IDと非PII集計のみ）をコードで強制。

## 8. セキュリティ / プライバシー / コンプライアンス

### 3つの信頼境界（最重要・最初に引く）
- **(A) 生体ゾーン**: 顔特徴量・顔画像は既存の顔認証/入退館ベンダー内に閉じ込め、**我々は一切受け取らない**。受領は「会員ID・店舗ID・入退館タイムスタンプ」の非生体イベントのみ。顔は個人識別符号＝漏洩しても再発行不能（顔は変えられない）。生体DBを持たない＝回復不能リスクを構造的に排除。ゾーン別占有はサーマル/mmWave（非生体）で取る。
- **(B) PIIゾーン**: 会員PII・決済はhacomonoを一次マスタとし、新スタックでは**Supabase1箇所にだけ**最小PIIを置く（RLSで「会員は自分の行だけ」をDB層強制）。`auth.js`の電話番号平文localStorage保存は本番化で最初に潰すseam。Supabase/Firebase AuthのOTPに委ね、自前でSMS認証・暗号化・総当たり対策を書かない。
- **(C) 分析ゾーン**: BigQueryには`member_sk = HMAC-SHA256(hacomono_cust_id, salt)`（saltはSecret Manager）の擬似匿名イベントのみ。PIIテーブルはBQへ複製禁止＝dbt sourceに定義せず、CIで`name/phone/face/birth`カラム混入を機械検査。

### 秘密管理（現アンチパターンの是正）
config.local.jsの平文Mapsキーは**git履歴未混入・未追跡を確認済（被害限定的）**だが、本番でクライアント配布JS（およびCapacitorバンドル）に乗れば閲覧可能。対処: ①即ローテーション、②HTTPリファラ制限（本番/staging/localhost）＋Maps JavaScript APIのみ＋日次クォータ上限の三重防御、③隠せる用途（Geocoding等）はBFFプロキシ経由、④Web版とアプリ版で鍵を分離（アプリはBundle ID/署名制限のネイティブ鍵）。サーバ側秘密（DB資格情報・HMAC salt・hacomonoキー）はSecret Manager/Supabase Vaultに集約。

### 同意（consent.jsを活かす）
現状のオプトイン設計（既定オフ・明示的肯定操作）は良い。足すのは: ①**二層化**（「サービス提供」と「分析二次利用・第三者提供＝マチミル」を別オプトイン）、②サーバ側`consent_ledger`に保存し撤回を全ゾーンへ即時伝播、③撤回ログを監査可能に保持。顔認証の当初目的は「入退館認証」であり、経営/B2B転用は利用目的の追加同意が必要（APPI第17条）。同意の無い目的にはデータをmartへ流さないパイプライン分岐を実装。

### B2B再識別防御（多層）
松山の狭い商圏×小population（会員1.6万・81店舗の一部は来館少）では属性×時間帯交差で個人が浮く。stats.jsのk-匿名/差分プライバシーは現状「合成データの可視化」に過ぎないため、本番では実データに対し: ①k≥10未満セルは抑制(suppression)＋年齢は帯へ一般化、②Laplaceノイズ＋εバジェット管理、③**データクリーンルーム**（BigQuery Analytics Hub・集計クエリのみ・生レコードは外に出さない）、④契約で「再識別/名寄せ禁止・監査権・目的外利用禁止・連帯責任」、⑤合成でなく実分布でk値を定期再評価。dbt testで「count<kがclean room出力に残っていないか」を毎回検査。

### LLMへのPII漏出防止
LLMには代理IDと集計値のみ渡すプロンプト境界をコードで強制し、PII列をプロンプト構築層で機械除外。マネージドAPI（Claude）でも非PII原則を満たす。

### 「最小十分」の線引き（地方が現実に守れる範囲）
**やる**: PIIをSupabase1箇所に集約／生体非保有／同意2層＋撤回伝播／B2B多層匿名化＋クリーンルーム＋契約／Mapsリファラ制限／半期チェックリスト運用／APPI第26条の漏洩報告フロー雛形。
**やらない（過剰）**: 自前生体DB、自前KMS/HSM、24/7 SOC、フルISMS認証。規模に対し負荷過大で形骸化を招く。

## 付録A. iOS/モバイル戦略（既存クーポンテンプレ公式アプリ「P・SPO24」の置き換え・移行）

**要旨**: 地方・小規模・既存hacomono/顔認証という制約下では、デモPWA資産（12HTML/22JS/5層CSS=約7,185行）を捨てずに最大限活かす Capacitor ラッパー方式が唯一の現実解。素のES Modules + no-buildで書かれているため、Capacitorのwebディレクトリにほぼそのまま載り、ネイティブで欠けるプッシュ通知・Apple/Google Walletの会員証・顔認証ローカル連携・確実なオフラインだけをプラグインで足す。既存OEMアプリ「P・SPO24」(iOS id1499960318 / Android com.couponapp2.chain.tac12785)はベンダーアカウント所有のため同一IDの引き継ぎは不可前提で、自社Apple/Google開発者アカウントを新規取得し新規アプリとして出し、旧アプリは「移行バナー＋サポート終了告知」で計画的に退役させる。Swiftネイティブやフルクロスプラットフォーム(RN/Flutter)は、地方の学生BPO＋AI実装者という体制では工数・保守が破綻するため非推奨。


**推奨**: Capacitor（@capacitor/ios + @capacitor/android）でデモPWAをラップし、自社新規ストアアカウントから新アプリとして公開。コア体験はWebView内の既存PWA、ネイティブ依存だけプラグイン化：プッシュ=@capacitor/push-notifications + Firebase Cloud Messaging（FCM/APNs）、会員証=PassKit(.pkpass)/Google Wallet API、確実なオフライン=Capacitorの同梱アセット（Service Worker任せにしない）、顔認証＝既に入退館ゲート側で取得済みのデータをhacomono/中間APIから引く方式（アプリ内で顔照合はしない）。配信は将来的に @capacitor/live-updates 風のOTAでWeb層だけ無審査更新できる構成にし、地方小規模でも継続改善できるようにする。1年Phase1のMVPはこのラッパー1本でiOS/Android同時に出す。


### 意思決定

- **Q: アプリ実装方式：Swiftネイティブ / RN・Flutter / PWA+Capacitorラッパー のどれにするか**
  - 選択肢: Swift（+Kotlin）フルネイティブで作り直す
  - 選択肢: React Native または Flutter で書き直す
  - 選択肢: 既存PWAをCapacitorでラップしWebView配信（ネイティブ機能のみプラグイン）
  - → **推奨**: Capacitorラッパーを強く推奨。デモは既に no-build の素ES Modules（HTMLが<script type="module">で直import、ui.js:298でsw.js登録済み）で書かれており、Capacitorの www/ にコピーするだけで起動する。約7,185行のUI資産・5層CSS・12ページがほぼ無改修で再利用でき、追加実装はプッシュ/Wallet/オフラインの薄いプラグイン層のみ。Swiftネイティブは体験は最高だがiOS/Android二重実装＋継続保守を学生BPO＋AI実装者では1年で回せない。RN/FlutterはこのデモのDOM直書きUIを全部書き直す＝デモ資産が死ぬため、投資対効果が最悪。地方小規模の現実制約に最も合致するのはCapacitor。
- **Q: 既存ストアID（iOS id1499960318 / Android com.couponapp2.chain.tac12785）を引き継ぐか、新規で出すか**
  - 選択肢: OEMベンダーから既存アプリ枠の譲渡（App Store/Play移管）を交渉し同一IDを継承
  - 選択肢: 自社開発者アカウントを新規取得し新アプリとして公開、旧アプリは退役
  - 選択肢: 旧アプリを残しつつ新アプリを併存（当面2本）
  - → **推奨**: 自社新規アカウント＋新規アプリを推奨（現実的かつ低リスク）。com.couponapp2.chain.tac12785 というpackage名はテンプレOEM(couponapp2)ベンダー所有で、アプリ移管(App Transfer)はベンダーの協力と契約解消が前提となり、地方事業者が短期で握るのは交渉コストが高い。よって最初から自社Apple Developer Program（法人=株式会社P・SPOカンパニーのD-U-N-S番号で登録）とGoogle Play デベロッパー法人アカウントを取得し、独自Bundle ID（例 jp.pspo.app）で新規公開。既存アプリは退役（移行告知バナー→数ヶ月後にサポート終了）。ただし旧アプリのレビュー資産・既存ユーザーへのリーチは惜しいので、退役は『新アプリ十分配布後』に段階実施する。
- **Q: 顔認証データをアプリ内でどう扱うか（アプリで顔照合するか、しないか）**
  - 選択肢: アプリでカメラ→顔認証を実装し入館もアプリで行う
  - 選択肢: 入館は既存の顔認証ゲートのまま、アプリは来館タイムスタンプ等の結果データだけ受け取る
  - 選択肢: 顔認証は一切扱わずQR/会員証のみ
  - → **推奨**: 『入館ゲートは現状維持、アプリは結果データ（来館タイムスタンプ）を受け取るだけ』を推奨。デモの非PII・カメラレス設計（README/consent.js/member.jsが顔・個票を持たない方針を明記）と完全に一致し、APPI上もアプリ側に生体情報を持ち込まない方が安全。顔データは既存ゲート/hacomono側に留め、アプリは中間API経由でランク・連続記録・利用連動割引（member.jsのロジック）に変換して『還元』を見せる。アプリでの顔照合実装はストア審査(生体)・法務・実装コストが跳ね上がり、1年Phase1では非推奨。
- **Q: オフラインとデータ更新をService WorkerとネイティブのどちらでやるかOTA更新を入れるか**
  - 選択肢: 現状のService Worker(sw.js)のみに依存
  - 選択肢: Capacitorの同梱アセット＋ネイティブストレージで確実オフライン化し、Web層はOTAで更新
  - 選択肢: 毎回ストア審査で全更新
  - → **推奨**: 『同梱アセットで確実オフライン＋Web層OTA』を推奨。WKWebView/Android WebViewではService Workerの挙動・キャッシュ寿命が不安定なため、sw.js(network-first)単独では起動保証が弱い。Capacitorはwww/をアプリ本体に同梱するので初回からオフライン起動が保証される。さらにLive-Updates系（Capacitor公式のサブスク版、またはOSSのcapacitor-updater）でWeb層(HTML/CSS/JS)だけストア無審査で配信できるようにすると、地方小規模でもバグ修正・データ更新（insights.js等）を当日反映でき、ストア審査待ちの停滞を回避できる。ネイティブプラグインを足すときだけストア審査、という運用にする。

## 結論：なぜ Capacitor ラッパー一択か（コードに基づく根拠）

このデモは「Capacitorで包むために最適化されたかのような」構造をしている。確認できた事実：

- **no-build の素ES Modules**：`index.html:171`〜、`me.html:90`〜、`app.html:65`〜 がいずれも `<script type="module">` で `./js/*.js` を直importしている。トランスパイル・バンドラ不要なので、Capacitorの `webDir` にディレクトリごと置けば**そのまま動く**。RN/Flutterに移すと、このDOM直書きUI（`js/ui.js` の `c-tabbar` 下部ナビ、`renderMap`、`renderServiceCatalog`等）を全部書き直す羽目になり、約7,185行の資産（HTML+JS+CSS実測）が無に帰す。
- **外部fetch皆無・全static**：`grep fetch(` の結果、ネットワーク呼び出しは `navigator.serviceWorker.register("sw.js")`（`js/ui.js:298`）と `navigator.geolocation`（`js/widgets.js:108`, `js/facilities.js:56`）のみ。データは `insights.js`/`stores-real.js`/`geo.js`/`data.js` に静的埋め込み。つまり**ネイティブ化で壊れるネットワーク依存が存在しない**＝移植リスクが極小。
- **SW登録済み・PWA設計済み**：`manifest.webmanifest` に `display:standalone`・`shortcuts`（live/me/growth）まで定義済み。PWAとして既に「ホーム画面に追加」体験が成立しており、Capacitorはその延長線。

逆に、Swiftネイティブ／RN・Flutterは**地方小規模＋学生BPO＋AI実装者**という体制で1年Phase1を回すには重すぎる。Swiftはi/Android二重保守、RN/Flutterはデモ資産の全書き直し。**現実制約に対する唯一の合理解がCapacitor**。

## アプリの「面」分割（12ページをどう振り分けるか）

READMEとコードが既に「顧客データと経営データは別系統」と設計している。これをアプリ境界に落とす：

| 面 | ページ | 配信形態 |
|---|---|---|
| **モバイルアプリ（Capacitor, 一般会員）** | `app`(会員日常)/`area`(店舗・空き)/`live`(混雑ライブ)/`me`(マイページ・会員証)/`price`(料金)/`guide`(入会導線)/`index`(集客LP) | アプリ本体WebView＋ネイティブプラグイン |
| **管理Web（アプリに入れない）** | `growth`(経営AIダッシュボード)/`data`(データ戦略)/`machimiru`(B2B人流)/`brief`(経営層サマリ)/`critique`(自己批評) | 管理者ログインのWebアプリ。一般ストアアプリに混ぜると審査・情報設計が崩れる |

`guide`/`price` の hacomono 導線（`index.html:197` `OFFICIAL.hacomono`、`guide.html:64`、`price.html:95`）は、`critique.html:74` が自ら指摘する「外部ブラウザに蹴り出される」痛点なので、**Capacitor Browser（SFSafariViewController/Custom Tabs）でアプリ内表示**にして体験分断を解消する。これがそのまま「テンプレOEMより良い」という提案の証明になる。

## 4. レンダリング選定の根拠（SSR/SSG/SPA）

- **SSR全面は不採用**: SSRはサーバ常駐＝障害点・コスト・運用負債。地方小規模体制＋「すでに静的で動いている」資産に対して過剰。
- **集客=SSG**: 唯一SEOが事業価値に直結（新規会員の入口）。Astroなら現index.htmlの構造をほぼそのまま .astro 化でき、islandsで reels/visuals の動的部分だけJS化。robots を Disallow→Allow に変えて初めて集客サイトになる（現状 robots.txt は全Disallow）。
- **会員=SPA/PWA**: PWAインストール体験・オフライン・hacomono/顔認証連携が要件。現 no-build PWA を当面維持できるのが強み（破壊的移行を避けられる）。SEO不要なのでSSR不要。
- **社内=認証SPA**: SEOゼロ・認証最優先。レンダリング方式は重要でない。Access の内側に置けば素のSPAで十分。

## 5. 認証境界の具体

- **会員(app.pspo.jp)**: hacomono を master of record にする（会員台帳・予約・決済が既にそこにある＝二重持ち禁止）。hacomonoがSSO/APIを提供するならそれを起点に、顔認証の来館イベント(タイムスタンプ取得済)を会員IDで紐付け。提供がなければ電話番号OTP最小実装＋予約はhacomonoへdeep link。`auth.js` の localStorage 擬似(L9–17)は本番では撤去。`consent.js`(オプトイン同意)は良い実装なので会員面に残し、同意状態はサーバ同期。
- **社内(ops.pspo.jp)**: Cloudflare Access + Google Workspace SSO。既存の会社メール基盤でゼロ追加コスト・即日運用可。IP/VPN制限を併用。これが現状最大リスク(growth/data素HTML公開)への直接の解。
- **B2B(machimiru)**: Clerk等マネージドIdP（契約企業の招待制・将来のSSO拡張に強い）。Phase4マーケットプレイスのAPIキー発行基盤にもなる。

## 付録C. バックエンド/データ基盤（API設計・認証連携・来館イベントパイプライン・同意管理・DWH/レイクハウス・PII分離クリーンルーム）

**要旨**: デモはデータを静的JSへ埋め込み（バックエンド不在）で、insights.js(40KB分析)・stores-real.js(81店舗)・geo.js(地理)・member.js(localStorage会員)が「本来サーバが持つべきもの」を仮置きしている。本格運用はSupabase(マネージドPostgres)中心の薄いBFF＋BigQueryクリーンルームを推奨し、hacomono(契約/予約/課金)と顔認証ゲート(来館)を「正本=System of Record」として残したまま、来館イベントを非PII射影してDWHへ集約する。地方小規模・非PII原則・既存hacomonoという与件では、自前で重いマイクロサービス群やKafka/Snowflakeを建てるのは過剰で、BFF(薄いREST)＋イベント収集Functions＋BigQueryの3層が現実解。最大の設計上の核心は、PII(顔特徴量・氏名・電話)を持つ営業系ゾーンと、顧客アプリ/B2Bマーケットプレイスが触る非PII分析ゾーンをデータ層で物理分離し、両者をsalted-hashの会員擬似IDだけで結ぶクリーンルーム境界を「最初に」引くこと。これを後から足すと作り直しになる。


**推奨**: マネージド3層 + クリーンルーム境界。(1)アプリBFF/正本: Supabase(Postgres + Auth + Row Level Security + Edge Functions)。会員アプリの読み書き・consent台帳・member派生値を持つ。hacomonoが契約/課金/予約の正本、顔認証ゲートが来館の正本で、Supabaseはそれらの「非PII射影のミラー＋アプリ独自状態」に徹する。(2)イベント収集パイプライン: 顔認証ゲート/hacomono → Webhook/CSVバッチ → Supabase Edge Function(ingest)で受信 → 即座にPII分離(顔特徴量は破棄、生member_idはHMAC-SHA256でpseudonym化) → events_raw(Postgres)とObject Storage(生イベントのParquet)の両方へ。(3)DWH/クリーンルーム: Google BigQuery(オンデマンド課金=使った分だけ。地方小規模に最適)に非PII射影のみ日次ロード。dbtでstaging→mart(cohort/churn/occupancy/人流mart)を構築。B2B「マチミル」とgrowthダッシュボードはBigQueryのmartだけを参照し、PII正本(Supabaseのmembers_pii)には到達できない。API様式はGraphQLでなくREST + OpenAPI(学生BPO+AI実装者が保守でき、Supabase自動生成RESTと相性良)。認証は会員=hacomono OIDC/SSO委譲(自前パスワードを持たない)、社内=Google Workspace SSO + BigQuery IAM、B2B=APIキー+スコープ。js/config.local.jsのMapsキーに代表される「秘密をgitignoreで隠す」アンチパターンは、Supabase Vault / Google Secret Manager + サーバ側プロキシへ移す。


### 意思決定

- **Q: DWH/レイクハウスの製品選定: BigQuery(GCP) か Snowflake か Supabase Postgres内で完結か**
  - 選択肢: BigQuery オンデマンド(スキャン課金)
  - 選択肢: Snowflake(仮想ウェアハウス課金)
  - 選択肢: Supabase/Postgres + dbtで分析もそのまま
  - 選択肢: Databricks レイクハウス
  - → **推奨**: BigQuery オンデマンドを推奨。理由: (1)地方小規模で『使った分だけ』のスキャン課金がアイドルコスト最小(夜間に誰も触らない時間が長いジム経営ダッシュボードに最適、Snowflakeの常時ウェアハウスは過剰)。(2)BigQuery Data Clean Rooms / Analytics Hub が非PII共有の標準機能を持ち、insights.jsのCIVIC_APPS/DATA_ANALOGSが想定するB2Bマチミルの第二収益線(他社へのデータ提供)を法定の匿名加工境界つきで実装できる。(3)会員1.6万・81店舗・来館は1日数千件規模でデータ量はTB級でなくGB級——Databricks/Snowflakeのフルパワーは不要。トレードオフ: ストアド処理よりスキャンコスト管理(パーティション/クラスタリング/予約スロット上限)の運用規律が要るが、dbt+パーティション設計で吸収できる。
- **Q: PIIと非PIIの分離をどこで引くか(クリーンルーム境界の実装位置)**
  - 選択肢: アプリケーション層のみで分離(同一DBにPII/非PII両方)
  - 選択肢: 別データベース/別プロジェクトで物理分離 + 擬似ID結合
  - 選択肢: BigQuery Data Clean Roomsの集計専用ビューのみで論理分離
  - 選択肢: 全部1テーブルでカラム権限(列レベルセキュリティ)
  - → **推奨**: 別プロジェクト物理分離 + 擬似ID結合を推奨。members_pii(氏名/電話/hacomono顧客ID/顔認証ベンダID)はSupabaseの分離スキーマ(RLS厳格・社内最小権限のみ)に置き、BigQueryへは一切複製しない。BigQuery側はmember_sk(=HMAC-SHA256(member_id, ピンに保管したsalt))だけを持つ。再識別が必要な業務(離反会員への先回り特典送付)は、BigQueryで対象member_skリストを出す→Supabase側の権限者だけがsk→PII逆引きしてhacomono/LINEへ配信、という二段で人間の権限を挟む(insights.js GOVERNANCE『human-in-the-loop』『最小権限と監査ログ』に整合)。論理分離(列権限)だけだと一つの権限事故でPIIが分析環境に漏れるため、物理分離が地方の小チームでも事故りにくい。
- **Q: 認証アーキテクチャ: 会員ログインの正本をどこに置くか(現auth.jsはlocalStorage擬似ログイン)**
  - 選択肢: Supabase Authを正本にして独自ID体系
  - 選択肢: hacomono(pspo24.hacomono.jp)にOIDC委譲しSupabaseはミラー
  - 選択肢: 顔認証ベンダのIDを正本
  - 選択肢: Firebase Auth
  - → **推奨**: hacomonoにOIDC/SSO委譲を推奨。理由: 会員の契約・課金・予約の正本は既にhacomono、来館の正本は顔認証ゲート。ここでSupabase Authに独自パスワードを持たせると二重管理・退会反映漏れ・PII増殖になる。hacomonoがOIDCを提供すればそれをIdPに、無ければhacomono顧客番号+生年月日等での突合 or マジックリンク(SMS OTP)でアプリ独自セッションを発行し、内部でhacomono_customer_id↔member_skをマップする最小テーブルだけ持つ。auth.jsが訴求する『外部ブラウザに蹴り出されない1タップ』UXは、PWA内でトークン取得を完結させるOIDC PKCEフローで維持できる。トレードオフ: hacomonoのOIDC対応可否に依存——未対応なら当面はSMS OTP(Supabase Auth phone) + 後でhacomono台帳と日次突合する折衷で進める。

### リスクと緩和

- 顔認証ベンダのデータ連携仕様が不明/限定的: 来館タイムスタンプしか出ない or Webhookが無くCSV手動の可能性。緩和=最初はゲートの日次CSVバッチ取り込みで割り切り、リアルタイムは将来。ベンダ契約時に『非PII(member_id+timestamp+store_idのみ、顔特徴量は渡さない)エクスポート』を仕様に明記。
- hacomonoがOIDC/API連携に非対応または有料制限: 会員IDの正本突合ができないと擬似ID基盤が成立しない。緩和=当面SMS OTPでアプリ独自会員を作り、hacomono顧客番号は会員自己申告→日次CSV突合でマッピング。hacomono側のAPI/SSOオプション可否を早期にベンダ確認(クリティカルパス)。
- PII分離境界の運用が崩れる(便利さでBigQueryにPIIを入れてしまう): 一度混ざると匿名加工情報の前提が崩壊しB2B収益線が違法リスク化。緩和=dbtのsourceにPIIテーブルを定義しない+CIでPIIカラム名(name/phone/face)の混入を機械検査+四半期のデータマップ棚卸し。
- スキャンコスト/クエリ暴走でBigQuery請求が膨らむ: 地方予算では致命的。緩和=日付パーティション必須、SELECT *禁止のdbt規約、オンデマンドのスキャンバイト上限(maximum_bytes_billed)とプロジェクト予算アラート、ダッシュボードはmart集計済みテーブルのみ参照(生eventsを直接叩かせない)。

## 5. PII分離 / クリーンルーム方針(本レンズの核心)

1. **物理分離**: PII(members_pii)はSupabase `secure` スキーマに隔離。BigQuery(analytics)へは**絶対に複製しない**。dbtのsourceにPIIテーブルを定義せず、CIで`name/phone/face/birth`等のカラム名がanalytics側に混入していないか機械検査。
2. **擬似ID結合のみ**: 両ゾーンを繋ぐのは`member_sk = HMAC-SHA256(hacomono_cust_id, salt)`だけ。saltはSecret Managerに保管しEdge Functionでのみ使用。これによりBigQuery側だけ漏れても原本(氏名/電話/顔)へは戻れない。
3. **顔特徴量は受け取らない**: ingest時点で顔ベクトル/生映像は破棄し、`member_id + timestamp + store_id`の非PII射影だけを受ける(GOVERNANCE『非PII射影』『顔・個票を持たない』をデータ契約として顔認証ベンダ仕様に明記)。
4. **再識別の人間ゲート**: 離反会員への先回り特典(ANALYSES/AI_PLAYBOOK)は、BigQueryで対象`member_sk`を出す→Supabase権限者だけがsk→PII逆引きしてhacomono/LINE配信、の二段。自動でPIIに戻さない(human-in-the-loop)。
5. **B2B提供は匿名加工境界つき**: BigQuery Data Clean Rooms / Analytics Hubで、k-匿名(例 k>=10)未満のセルを抑制・丸め・差分プライバシーをmart層で強制。81店舗の一部は来館が少なく小population再識別リスクがあるため、dbt testで「count<k のセルがclean room出力に残っていないか」を毎回検査(consent.jsが画面で約束する『k-匿名化・差分プライバシー』を実装で裏打ち)。
6. **目的別同意のパイプライン分岐**: consent_ledgerのpurpose='b2b_provide'がgrantedな会員のみmachimiru martへ流す。同意撤回(オプトアウト)時は次回バッチでclean roomから除外。

## 6. 秘密管理の是正(Mapsキー問題)

`js/config.local.js`の平文キーは(a)即時ローテーション、(b)Google Cloudで HTTPリファラ制限 + Maps JavaScript APIのみ許可(config.jsのコメントが要求済み)、(c)できれば地図タイルもサーバ側プロキシ(Edge Function)経由にしてキーをクライアントへ出さない、へ。サーバ側の秘密(DB資格情報・HMAC salt・hacomono連携キー)はSupabase Vault / Google Secret Managerに集約し、コード・gitに一切置かない(GOVERNANCE『秘密はSecret Managerでコードに置かない』に整合)。

## 付録D. AI/MLプラットフォーム

**要旨**: デモは既に「答え」を出す統計（js/stats.js の生存時間分析・Huff・需要平準化・差分プライバシー）と、本番で使うべきモデル名リスト（js/insights.js の AI_PLAYBOOK）を持っており、本格運用は「ゼロから作る」ではなく「既存ロジックの教師信号を実データ（hacomono予約＋顔認証入退館タイムスタンプ）に接地する」移行である。地方小規模・非PII・既存hacomono/顔認証という制約下では、巨大MLOps基盤は過剰投資なので、最初の12ヶ月は『managed warehouse（BigQuery）＋dbt＋スケジュール実行のバッチML（Python/LightGBM/lifelines）＋マネージドLLM API（Claude/RAG）』に絞り、feature storeも専用製品でなくwarehouse上のテーブル規約で代替する。順序は「①離反予測（教師あり・既にCHURNで検証済み・LTV直撃）→②混雑/占有予測（オフピーク誘導の前提・ただし顔認証は入退館のみでゾーン占有は別途）→③クロスセル推薦→④行動系列の自己教師あり＆LLMエージェント」が現実解。LLMは社内BI（read-only SQLツール使用RAG）を先、会員サポートは後にし、コストはバッチ要約＋キャッシュで月数千〜2万円規模に収める。


**推奨**: 「Warehouse中心・バッチファースト・マネージドLLM」アーキテクチャを推奨。具体スタック: データ基盤=Google BigQuery（無料枠大・サーバレス・SQLだけで回る、地方小規模に最適）＋ dbt-core（変換とfeature定義をSQLバージョン管理）。取り込み=hacomono CSV/API・顔認証入退館ログ・アプリ行動ログを日次バッチで BigQuery へ（最初はGoogle Apps Script/Cloud Run jobsの cron で十分、Fivetran等の有償ETLは不要）。学習/推論=Python（LightGBM=離反/需要、lifelines=生存時間/卒業予測、Prophet or LightGBM=時系列占有、implicit/LightFM=推薦）を Cloud Run Jobs + Cloud Scheduler で日次バッチ実行し、スコアを BigQuery の predictions テーブルへ書き戻す→アプリ/hacomonoが読むだけ。リアルタイム推論サーバは当面作らない（バッチ日次スコアで離反/推薦は十分、占有だけ準リアルタイム）。feature store=専用製品（Feast等）を入れず、BigQuery上に features_member_daily / features_store_hourly を「学習と推論で同一SQL」で生成する規約で point-in-time整合を担保。モデルサービング=スコアはテーブル、占有予測のみ軽量API（Cloud Run・FastAPI）。MLOps=MLflow(セルフホスト or マネージド)でモデル/メトリクス記録、Evidently（OSS）で日次ドリフト監視、再学習は週次cron＋しきい値割れ時のSlack通知。LLM=Anthropic Claude API（Haiku=安価ルーティング/要約、Sonnet=BI推論）で、社内BIエージェントは「BigQueryのスキーマ＋メトリクス辞書をRAGし、read-onlyのSQL実行ツールとinsights集計ツールを使用」。会員サポートはFAQ＋hacomono手順をRAGする限定スコープで後発。全体の月額クラウド費は当初2万円以下を目標。


### 意思決定

- **Q: 最初に本番化するモデルは何か（離反予測 vs 占有予測）**
  - 選択肢: 離反予測を最優先（教師あり・既存CHURN検証済み・LTV直撃）
  - 選択肢: 占有予測を最優先（アプリのコア体験・オフピーク誘導の前提）
  - 選択肢: 2つ並行で着手
  - → **推奨**: 離反予測を最優先。理由(1)教師信号が既に存在＝CHURN(js/insights.js)に退会ラベル・在籍月・年齢/性別があり、追加センサ不要で入退館タイムスタンプ＋hacomono契約データだけで学習できる。(2)効果が金額で言える＝1人の退会防止LTV(月5,478円×平均在籍14ヶ月≒7.7万円)>新規CAC で経営層に刺さる。(3)占有のゾーン/設備粒度予測は顔認証が『入退館のみ』でゾーン占有データを持たない(insights.js currentCongestion/entry が明記)ため、まず店舗×時間帯の総量予測しかできず、設備別はYOLO等の追加投資が必要で初手には重い。占有は『店舗×時間帯の総来館予測』(入退館ログのみで可能)に絞ってPhase1後半に追加するのが正。
- **Q: feature store専用製品(Feast/Tecton)を入れるか**
  - 選択肢: Feast等のOSS feature storeを導入
  - 選択肢: BigQuery+dbtのテーブル規約で代替
  - 選択肢: Tecton等のマネージドSaaS
  - → **推奨**: BigQuery+dbtのテーブル規約で代替。理由：地方小規模・日次バッチが主体・オンライン低レイテンシ配信要件が当面ない段階で、Feastの運用(オンラインストアRedis等)はコスト/学習コストの過剰投資。学習用と推論用のfeatureを『同一のdbtモデル(features_member_daily)』から生成し、訓練時はas-of日付でフィルタ、推論時は最新日を読むことでtrain/servingスキューを防ぐ。会員サポートのオンラインパーソナライズが本当に低レイテンシで必要になった段階(Phase3〜)で初めてFeast+軽量KVを検討。
- **Q: LLMは自前OSSモデルかマネージドAPIか**
  - 選択肢: ローカルOSS LLM(Llama/Qwen等)を自前GPUで運用
  - 選択肢: マネージドAPI(Claude/Gemini)
  - 選択肢: ハイブリッド(機微処理だけローカル)
  - → **推奨**: マネージドAPI(Claude)を推奨。理由：体制が学生BPO＋AI実装者で専任MLOpsがおらず、GPUサーバ運用・量子化・推論最適化の固定費と人件費が地方小規模には重い。Claude Haikuでルーティング/要約、Sonnetで難しいBI推論、とコスト階層化すれば、社内BIは月利用が限定的なので数千〜2万円/月で収まる。PIIは『LLMに渡さない（非PII射影された集計値とスキーマのみRAG）』を設計原則にし、顔・個票はwarehouseのPII金庫から出さない＝マネージドAPIでもプライバシーを担保。将来コスト/データ量が膨らんだら、要約等の定型タスクだけ小型OSSをCloud Runに載せるハイブリッドへ。
- **Q: 占有のゾーン/設備別データをどう得るか（顔認証は入退館のみ）**
  - 選択肢: YOLO/RT-DETR等のカメラAIを全店導入
  - 選択肢: サーマル/レーダー(Butlr/Density型)の非カメラセンサ
  - 選択肢: 当面は入退館ログからの店舗総量予測に限定し、数店で混雑度を推定する弱教師でつなぐ
  - → **推奨**: 当面は入退館ログ由来の『店舗×時間帯の在館純人数』予測に限定し、設備別は導入しない。理由：insights.js が『顔認証＝入退館認証でゾーン別占有や滞在内訳は未取得』と明記しており、設備別精度を出すにはハード投資(カメラAI or サーマル)が必須。非PII/女性安心を最優先する同社方針(GOVERNANCE)とコストから、第一弾はカメラレスを維持し、入退館in/out差分で『今の在館人数』と時間帯予報を出す。設備別の需要が検証で固まった段階で、更衣室/サウナはButer型サーマル(非カメラ)、ジムフロアのみ頭部数YOLO(顔非保持)を1〜2店パイロット、という非PII優先の順で投資する。

### 実装手順

1. 【M0：データ基盤の最小立ち上げ(1-2ヶ月)】GCPプロジェクト作成→BigQueryに3スキーマ(raw/staging/marts)を作る。hacomono(予約・契約・会員属性)を日次CSVエクスポートかAPIでrawへ、顔認証入退館ログ(in/outタイムスタンプ・会員ID)をrawへ、PWAアプリ行動ログ(タップ/検索/チェックイン)をrawへ取り込む。PIIは別データセット(pii_vault)に分離し最小権限。Secret ManagerでAPIキー管理(現状config.local.jsのgitignoreはアンチパターン＝本番はSecret Managerへ移す)。
2. 【M1：dbtで変換とfeature定義(同月)】dbt-coreで staging→marts を構築。中核featureテーブル: features_member_daily(会員×日: 直近来館間隔・30日来館回数・利用業態数・最終来館からの経過日・在籍月・年齢/性別・プラン)、features_store_hourly(店舗×時刻: 在館純人数・曜日・天候等)。学習と推論で同一SQLを使い point-in-time整合を担保(train/servingスキュー防止)。
3. 【M2：離反予測を本番化(2-3ヶ月)】ラベル定義=『次30日以内に退会/休会』。学習= lifelines(Cox比例ハザード)で生存曲線＋ LightGBM で30日離反確率の2本立て(stats.jsのsurvivalCurveが既にプロトタイプ)。Cloud Run Jobs + Cloud Schedulerで日次バッチ→predictions_churn テーブルへ書き戻し。評価= churn AUC(目標0.75+)・PR-AUC・上位decileのlift。出力をhacomono/CRMやスタッフ画面に出し『危険会員へ先回り特典』をhuman-in-the-loopで発火。A/Bで介入効果(リテンション差)を測る。MLflowでモデル/メトリクス記録、Evidentlyで入力ドリフト監視。
4. 【M3：占有/需要予測(店舗総量)を追加(3-4ヶ月)】入退館in/out差分から『店舗×時刻の在館純人数』を作り、Prophet or LightGBM(曜日・時間・祝日・天候)で時間帯予報を学習→predictions_occupancy。stats.jsのlevelDemand(需要平準化)に接続し、オフピーク誘導クーポンの時間帯/割引を最適化(将来contextual banditへ)。占有のみ準リアルタイム需要があるので軽量Cloud Run API(FastAPI)で最新在館数を配信、設備別は当面出さない(非PII維持)。
5. 【M4：クロスセル推薦(4-6ヶ月)】1ID業態横断利用の共起から implicit/LightFM で未使用業態を推薦→predictions_reco。member.js のパーソナライズUIに接続し『多機能すぎて1施設しか使わない』割高感を解消。評価=推薦経由の新規業態利用率・利用業態数の増加。
6. 【M5：社内BI LLMエージェント(6-8ヶ月)】Claude(Haiku/Sonnet階層)で『スキーマ＋メトリクス辞書(insights.jsのANALYSES/DATA_SOURCESを起点)をRAG→read-only SQL実行ツール＋集計ツールを使用→自然言語で経営質問に回答』。PIIは渡さず非PII集計のみ。コストはキャッシュ＋日次サマリのバッチ生成で月数千〜2万円に抑える。growth.html/brief.htmlの経営ダッシュボードに統合。
7. 【M6：行動系列の自己教師あり＋会員サポートLLM(8-12ヶ月)】来館・業態・操作の系列で自己教師あり(次行動予測/シーケンス埋め込み)を学習し、離反/推薦のfeatureとして注入(性能底上げ)。会員サポートLLMはFAQ＋hacomono予約手順＋料金(price)をRAGする限定スコープで投入。卒業予測(lifelines、18歳離脱に直撃)を加え『4年プラン/卒業お祝い割』を先回り。
8. 【M7：MLOps定常化＋データ・ネットワーク効果(継続)】週次再学習cron＋しきい値割れ時Slack通知、Evidentlyでドリフト/データ品質を日次監視、MLflowでモデルレジストリ。『使う→シグナル増→個別最適/平準化→継続→使う』のフライホイール(insights.js DATA_FLYWHEEL思想)をKPI(利用業態数・来館頻度・離反率)で可視化。Phase2以降の他社のれん分け展開時に同一スキーマで横展開できるようdbt/特徴量を再利用可能に設計。

### リスクと緩和

- 顔認証は『入退館のみ』でゾーン/設備占有を持たない(insights.js currentCongestion/entry明記)→設備別占有予測は初手で出せない。緩和=第一弾は店舗総量の在館人数予測に限定し、設備別は検証後にサーマル(非カメラ)/頭部数YOLOを1-2店パイロットで段階投資。
- APPI(個人情報保護法)上、顔認証の同意目的は『入退館認証』であり離反予測/広告/第三者提供は別目的(GOVERNANCE/LIMITS明記)。緩和=会員価値還元(離反先回り特典・推薦)は本人便益で利用目的の追加同意を取りやすいが、データ販売/広告は匿名加工情報の法定基準＋オプトイン＋法務レビューを前提にゲートする。LLMにPIIを渡さない設計を徹底。
- 小population再識別リスク(松山・会員1.6万人で店舗×時間×属性の組合せ)→k-匿名でも再識別可能。緩和=stats.jsの差分プライバシー(Laplace)実装を本番のクエリ層に組み込み、k<閾値の集計は出さない・第三者提供はデータクリーンルーム/監査前提。
- データ量が小さく学習が不安定(月次退会数は数百人規模、CHURNでchurners6mo=3813)→過学習/不安定なAUC。緩和=シンプルなモデル(LightGBM/Cox)＋時系列CV＋較正(calibration)、決定はhuman-in-the-loop、効果はA/Bで検証してから自動化。
- 専任MLOpsがいない学生BPO体制で基盤が運用破綻するリスク。緩和=リアルタイム推論サーバ/Feast/Kubernetesを避け、BigQuery+dbt+Cloud Run cron+マネージドLLMという『SQLとPythonバッチだけで回る』最小構成に固定。監視はEvidently/MLflowのマネージド寄り構成で人手を最小化。
- LLMコストの暴走(社内BIで重いSonnet多用)。緩和=Haikuでルーティング/要約・Sonnetは難問のみ、結果キャッシュ、日次サマリのバッチ事前生成、月予算アラート。会員サポートLLMはスコープを料金/予約手順に限定し幻覚と費用を抑制。
- 現状のシークレット管理がアンチパターン(config.local.jsをgitignoreでしのいでいる)→本番でキー漏洩リスク。緩和=GCP Secret Manager/環境変数へ全面移行し、フロントにキーを置かない(Maps等はリファラ制限＋サーバ経由 or 制限付き公開キー)。

### 詳細

## 全体像：このプロジェクトは「ゼロから作るML基盤」ではない

根拠を先に示す。リポジトリには本格運用の設計図がほぼ揃っている。

- **モデル選定は確定済み** … `js/insights.js` の `AI_PLAYBOOK`（276-296行）に、各打ち手の OSS/手法が明記されている。離反=XGBoost/LightGBM、需要=Prophet/LightGBM、推薦=LightFM/implicit、RAG=sentence-transformers+LLM、卒業予測=lifelines(Cox/Kaplan-Meier)、出店=GeoPandas/H3+GBM、占有=YOLO/RT-DETR(頭部数のみ)。**本番化とは、このリストの手法を実データに接地して定期実行する作業**であって、技術選定の議論はほぼ終わっている。
- **統計ロジックは既にプロトタイプが動く** … `js/stats.js` に、生存時間分析(`survivalCurve`/`medianSurvival`/`retentionAt`/Cox整合の指数モデル)、需要平準化(`levelDemand`)、Huff出店モデル(`huffCapture`)、ハバーサイン距離、差分プライバシー(Laplaceノイズ)が**素のJSで実装済み**。これらは本番でPython(lifelines/LightGBM)に置き換える際の『正解の挙動』『教師信号』になる。
- **教師ラベルが既にある** … `js/insights.js` の `CHURN`（146-185行）に、在籍/退会のコホート（年齢・性別・在籍月・退会フラグ）が一次資料ベースで入っている。離反予測の学習に必要な構造が既に定義されている。
- **接地データの限界も明記済み** … `currentCongestion`/`entry`(30-31行)が『顔認証＝入退館認証で、ゾーン別占有や滞在内訳は未取得』と正直に書いている。これが「占有予測を初手にできない」理由の一次根拠。
- **ガバナンスの前提** … `GOVERNANCE`/`LIMITS`(322-366行)がAPPI・匿名加工・小population再識別・human-in-the-loopを明記。AI基盤の設計はこの制約を満たす形に固定される。

つまり**設計の仕事は「過剰投資せず、この既存ロジックを最小のマネージド基盤で定期実行に乗せる順序」を決めること**に尽きる。

### なぜこの構成か（トレードオフ）
- **BigQuery採用の理由**：サーバレスで運用ゼロ、SQLだけで回る、無料枠が大きい（月1TBクエリ・10GBストレージ無料）。会員1.6万人×日次イベントの規模なら**当初は事実上ほぼ無料〜数百円/月**。AlloyDB/自前Postgres+ワーカーより運用が軽い。Snowflakeは小規模には割高、Redshiftはクラスタ常時課金で不利。
- **リアルタイム推論サーバを作らない理由**：離反/推薦は『日次バッチでスコアをテーブルに書き、アプリは読むだけ』で体験上十分（離反確率は分単位で変わらない）。**唯一準リアルタイムが要るのは占有(在館人数)**なので、そこだけ軽量Cloud Run API。これでKubernetes/モデルサーバ(BentoML/KServe)の運用固定費を回避。
- **feature storeを専用製品にしない理由**：オンライン低レイテンシ配信要件が当面ないのにFeast(+Redis等オンラインストア)を運用するのは過剰。`features_member_daily`/`features_store_hourly` をdbtで一元定義し、**学習時はas-of日付でフィルタ、推論時は最新日**を読むことでpoint-in-time整合とtrain/servingスキュー防止を実現できる。専用FSが本当に要るのは「会員サポートLLMがオンラインで個人特徴を引く」フェーズ(Phase3+)。

### ① 離反予測（最優先・教師あり）
- **ラベル**：観測日から次30日以内の退会/休会=1。`CHURN`構造(年齢/性別/在籍月/退会)が既に教師信号の形。

### ② 占有/需要予測（店舗総量・カメラレス維持）

## 付録E. レンズ5: リポジトリ / ブランチ / CI-CD / 環境

**要旨**: 現状はGitHub Pages（branch公開・CIなし）で1リポ1ドメインに全機能を集約した健全な単一リポジトリだが、(1) $HOME/.git に別アカウント(masuhama1946/git-pracitce)の空リポが残存し git add 暴発リスクがある、(2) Maps APIキーが config.local.js 経由でクライアントへ実配信される＝gitignoreは履歴しか守らない、(3) CI/CDもfeature flagも環境分離も無い、という3点が本番化の障害。地方・小規模・学生BPO＋AI実装者という体制ではポリレポの運用コストは負債になる。よってGitHub Actionsを単一CIとした pnpm + Turborepo のモジュラーモノレポへ移行し、trunkベース（短命feature/PR必須・main保護）＋ dev/staging/prod の3環境をVercelのPreview/Production機構で吸収し、デモは別オーガニゼーション/別ドメインに凍結保存して本番と分離する。既存Playwright 25×2を本番CIのPRゲートにそのまま昇格させる。


**推奨**: モジュラーモノレポ（pnpm workspaces + Turborepo）に統合し、ホスティング/CIは GitHub Actions（PRゲート）+ Vercel（環境別デプロイ）。ブランチは trunk-based（保護されたmain＋短命feat/fix＋必須PRレビュー1名＋必須CI green）。環境は dev=各自ローカル / staging=Vercel Preview（固定staging環境変数）/ prod=Vercel Production。シークレットは Vercel Env と GitHub Encrypted Secrets に集約し、クライアント露出が不可避なMapsキーはコードから外しビルド時注入＋Google CloudのHTTPリファラ制限を本番ドメインに限定。feature flagは専用SaaS不要で config パッケージの環境変数駆動フラグ。デモ現リポは proposal-archive org へ移管しPagesのまま静的凍結、本番は新org pspo-company 配下の新規モノレポ。$HOME/.git の stray repo は削除しグローバルガードを敷く。


### 意思決定

- **Q: 会員アプリ/Web/B2B/バックエンド/MLを monorepo で束ねるか polyrepo に分けるか**
  - 選択肢: pnpm + Turborepo のモジュラーモノレポ（apps/* + packages/*）
  - 選択肢: 機能別ポリレポ（app/web/api/ml を別リポ）
  - 選択肢: 現状維持の単一フラットリポ（パッケージ分割なし）
  - → **推奨**: モノレポを強く推奨。根拠: 体制が学生BPO＋AI実装者の小規模で、ポリレポはCI設定・TypeScript型・hacomono/顔認証クライアント・店舗マスタ(stores-real.js, geo.js, insights.js)の共有を複数リポに分散させ同期事故を生む。モノレポなら packages/config(しきい値・LEVELS・ブランド)・packages/data-model(店舗/会員/占有スキーマ)・packages/api-client(hacomono/顔認証ラッパ)を1箇所で型付き共有でき、Turborepoのリモートキャッシュで小さなCI時間に収まる。MLだけはPython資産が重いので apps/ml をモノレポ内に同居させつつ独立Pipfile/uv管理（言語混在でもタスク境界はTurboで切る）。
- **Q: git ブランチ戦略は trunk-based か git-flow か**
  - 選択肢: trunk-based（main直結・短命feature・PR必須・タグでリリース）
  - 選択肢: git-flow（main/develop/release/hotfix の多ブランチ）
  - 選択肢: GitHub Flow に staging long-lived を1本追加
  - → **推奨**: trunk-based + 保護main + staging長命1本（実質GitHub Flowの拡張）を推奨。根拠: develop/release の二重統合は同時並行が常時2-3人の規模に対し過剰で、マージ待ち・コンフリクト解消の人的コストが見合わない。mainを保護（直push/force-push禁止・必須PRレビュー1名・必須CIステータスチェック・linear history）し、feat/* fix/* は数日で閉じる短命運用。リリースは vX.Y.Z タグで、stagingはQA用にmainを追従する単一の長命ブランチのみ許可。
- **Q: ホスティングはGitHub Pages継続か別基盤へ移すか**
  - 選択肢: Vercel（Preview=staging / Production=prod、Edge FunctionsでBFF同居）
  - 選択肢: GitHub Pages継続＋Actionsで静的ビルド
  - 選択肢: Cloudflare Pages/Workers
  - → **推奨**: Vercelを推奨（Cloudflare Pagesが次点）。根拠: 攻め=データ×AIではレコメンド/LLMエージェント用のサーバ関数とBFFが必須になり、Pagesの『branchから静的配信』ではPreview環境・サーバ関数・環境別シークレットを賄えない。VercelはPR毎にPreview URLを自動発行しstaging分離を無料〜低コストで実現、本番ドメイン割当も容易。ただしML推論やhacomono連携の重い処理は将来Cloud Run等へ逃がす前提（VercelはBFF/エッジに限定しベンダーロックを浅く保つ）。
- **Q: デモは本番化後も継続するか、本番は別オーガニゼーションか**
  - 選択肢: デモを別org(proposal-archive)へ移管しPagesで静的凍結、本番は新org pspo-companyで新規
  - 選択肢: 現リポをそのまま本番に発展させる
  - 選択肢: デモを削除し本番に一本化
  - → **推奨**: デモは別org/別ドメインへ凍結保存し、本番は新org pspo-company配下の新規モノレポを推奨。根拠: デモ(masuhama2147-svg/pspo-company-demo)は『P・SPO24置き換え体験』の提案アーカイブとして価値が残り、URL(masuhama2147-svg.github.io/pspo-company-demo)を保ったまま凍結すると関係者が再参照できる。本番を別orgにすることで権限・Secrets・課金・ドメインを会社資産として三福HD/P・SPOカンパニー名義に正しく帰属させられ、益浜個人アカウント依存を断てる。

### リスクと緩和

- [ベンダーロックイン] Vercel依存が深まるとML推論やhacomono大量連携でコスト/制約が出る → BFF/エッジ用途に限定し、重い処理(ML推論・バッチ)はコンテナ(Cloud Run等)へ逃がせるよう packages/api-client を基盤非依存に保つ。最悪Cloudflare Pages/Workersへ移行できる薄い結合を維持。

## 付録F. セキュリティ / プライバシー / コンプライアンス

**要旨**: 現状デモは「非PII・カメラレス・全クライアント完結」で、プライバシー上の地雷を構造的に避けている健全な出発点だが、本格運用への移行は「初めてPIIとサーバーが登場する瞬間」であり、そこに統制を集中させるべきである。最大の論点は3つ——(1) 顔認証由来の来館データを会員に還元するには生体情報そのものを我々のスタックに持ち込まず hacomono/入退館システム側に隔離し『顔特徴量→会員ID→タイムスタンプ』だけを受け取る分界、(2) B2B/マーケットプレイス提供は「店舗×時間×属性」の組合せで小population再識別が起きるため、k-匿名化＋差分プライバシー＋データクリーンルーム＋契約・監査を多層で重ねること、(3) config.local.js の平文Maps鍵は幸い git 履歴に未混入（検証済み）で被害は限定的だが、本番では鍵はバックエンド経由かHTTPリファラ制限必須。地方小規模体制で守れる「最小十分」は、PIIを集める場所を1箇所(マネージドBaaS)に閉じ込め、生体・第三者提供だけ厳格運用し、残りは非PII射影を貫くことである。


**推奨**: 【データ分界アーキテクチャ＝最重要】3つの信頼境界を引く。(A) 生体ゾーン: 顔特徴量・顔画像は既存の入退館/顔認証ベンダー内に閉じ込め、我々は一切受け取らない。受領するのは『会員ID・店舗ID・入退館タイムスタンプ』の非生体イベントのみ。(B) PIIゾーン: 会員の電話/氏名/決済は hacomono を一次マスタとし、我々の新スタックでは Supabase(Auth+Postgres+RLS, 東京リージョン) 1箇所にだけPIIを置く。auth.js のSMS OTPは Supabase Auth / Firebase Auth に置換（自前で電話番号を平文localStorageに残さない）。(C) 分析ゾーン: BigQuery/dbt 上で『会員ID→ランダム化代理ID(ソルト付きHMAC, ソルトはSecrets Manager)』に擬似匿名化した非PIIイベントのみを置き、ここでAI/レコメンド/人流を計算。B2B『マチミル』に出るのはこの分析ゾーンから k-匿名化(店舗×時間×属性で最小群k>=10を強制)＋差分プライバシー(Laplace, ε管理)＋集計のみ。秘密管理は GitHub Pages→管理SaaS(Cloudflare Pages/Vercel)へ移し env var + Maps鍵はリファラ＆API制限。同意は consent.js のオプトイン設計を活かしつつ、分析二次利用と第三者提供を別同意に分離し、撤回ログを残す。地方体制の現実解として『自前インフラを建てず、Supabase + Cloudflare + BigQuery のマネージド3点セット＋IaaSではなく PaaS』で運用負荷を最小化する。

1. 【Phase0・即時/今週】秘密と生PIIの止血。git履歴を再確認のうえ(検証済み: Maps鍵は履歴未混入・生データCSVも未追跡)、Google Cloud Console で当該Maps鍵に『HTTPリファラ制限(本番ドメイン＋localhost)』『Maps JavaScript APIのみ許可』『日次クォータ上限』を設定。生データ/配下の 名刺データ.csv・PSPO____________.csv 等(実PII)は .gitignore 済みを再確認し、開発端末から暗号化保管(または削除)。漏洩時の鍵ローテ手順を1枚にメモ化。
2. 【Phase1・MVP基盤】PIIの単一集約点を作る。Supabase(東京リージョン, Auth＋Postgres＋Row Level Security)を採番。auth.js の擬似ログイン(電話番号を平文localStorage保存)を Supabase Auth の SMS/パスキーOTPへ置換し、PWAは『最小限のセッショントークン＋擬似匿名の代理ID』だけをローカルに持つ。決済・本人確認は hacomono に残し、SSO/OIDC連携キーで紐付け。consent.js を活かして同意状態をサーバ側(consentテーブル)にも保存し、撤回(オプトアウト)を即時反映・監査可能にする。
3. 【Phase1・データ分界】顔認証の生体ゾーンを隔離する契約・連携を確定。入退館/顔認証ベンダーと『生体(顔特徴量・画像)は我々に渡さず、会員ID＋店舗ID＋入退館タイムスタンプの非生体イベントのみ連携』を文書化。受領イベントは ingest 時に 会員ID→HMAC-SHA256(ソルトはSecrets Managerで管理・定期ローテ)で擬似匿名化し、分析ゾーン(BigQuery)へ。生体は一切スタックに入れない。
4. 【Phase1・統制の地ならし】最小十分なガバナンス文書を整備。(1)プライバシーポリシー＝利用目的の特定(APPI第17条)・第三者提供の有無・委託先(hacomono/Supabase/Maps)・保有期間・開示等請求の窓口。(2)同意の二層化＝『サービス提供のための利用』と『分析二次利用・第三者提供(マチミル)』を別オプトインに分離。(3)データマッピング(どのデータがどこに・誰に流れるか)を1枚に。(4)漏洩時のインシデント対応・個人情報保護委員会への報告フロー(APPI第26条, 報告義務)を雛形化。
5. 【Phase2・AI/分析ゾーン】BigQuery＋dbt で非PIIイベントの分析基盤を構築。離反/需要/占有予測・レコメンド・LLMエージェントは、すべて擬似匿名化済みデータ上で動かす。LLMへ会員PIIや電話番号を渡さない(プロンプトに代理IDのみ)プロンプト境界をコードで強制。アクセスログ・監査ログ(誰がどのデータに触れたか)をBigQuery/Supabaseで保持。
6. 【Phase3-4・B2B/マーケットプレイス】マチミル提供パイプラインに匿名加工を強制実装。dbtモデルで『店舗×時間帯×属性の最小群 k>=10 未満セルは抑制(suppress)、年齢は帯へ一般化、カウントはLaplaceノイズ(εバジェット管理)』を出力前ゲートとして必須化(stats.js のロジックを本番データへ移植・検証)。提供形態は第一にデータクリーンルーム(BigQuery Analytics Hub/共有データセットで集計クエリのみ許可、生レコードは外に出さない)。提供契約に『再識別・名寄せ・他データ突合の禁止』『監査権』『目的外利用禁止』『漏洩時の連帯責任』を明記。匿名加工情報として第三者提供する場合は APPI の作成基準・公表義務・識別行為の禁止を遵守。
7. 【継続】半期ごとの軽量レビュー。再識別リスクの実データ評価(合成でなく実分布でk値を再計算)、同意撤回が全ゾーンに伝播しているかの確認、Maps鍵/HMACソルトのローテ、委託先(Supabase/hacomono/顔認証ベンダー)の安全管理体制の確認。学生BPO＋AI実装体制で回せるよう、チェックリスト化して属人化を避ける。

### リスクと緩和

- 【生体情報の機微性】顔認証データ(個人識別符号)は漏洩しても本人が顔を変えられず再発行不能＝被害が永続。緩和: 案A(生体非保有)を徹底し、生体はベンダー閉域に隔離。万一にも顔特徴量を自社で持たない契約・データフローにする。
- 【小population再識別】松山の狭い商圏＋会員数の少ない店舗では、店舗×時間×属性の交差で個人が特定され得る。マチミルの『個人は追跡しない』という約束(machimiru.html)が破れると事業の信頼基盤が崩壊。緩和: k>=10抑制＋差分プライバシー＋クリーンルーム＋契約の多層防御。合成データでなく実分布でk値を再評価。
- 【秘密の露出】現状 config.local.js は平文Maps鍵。検証では git履歴・生データCSVへの混入は無し(被害限定的)だが、本番でクライアント配布JSに鍵を置けば誰でも閲覧可能。緩和: リファラ＆API＆クォータ制限を必須化、理想はバックエンドプロキシ経由。鍵ローテ手順を常備。
- 【同意と二次利用の乖離】サービス提供同意のまま分析・第三者提供へ流用するとAPPI(利用目的の特定・変更)違反かつ倫理違反。緩和: 同意を二層化し、撤回を全ゾーンへ即時伝播。撤回ログを監査可能に保持。
- 【委託先・連鎖リスク】hacomono/Supabase/顔認証ベンダー/Maps と委託先が増えるほど安全管理措置(APPI)と監査責任が拡大。緩和: PIIの保管点をSupabase1箇所に最小化し、委託先一覧と安全管理体制の定期確認をチェックリスト化。
- 【LLMエージェントへのPII漏出】データ×AIの目玉であるLLMエージェントにPII(電話番号・氏名)がプロンプトとして渡ると、外部API送信・ログ残存で漏洩。緩和: LLMには代理IDと集計値のみ渡すプロンプト境界をコードで強制し、PII列をプロンプト構築層で機械的に除外。
- 【統制の形骸化(地方小規模体制)】文書だけ作って運用されないリスク。緩和: 監査・ローテ・k値再評価を半期チェックリスト化し、学生BPO＋AI実装体制で回せる最小手順に落とす。完璧主義でなく『最小十分』を明示。

## 1. 12ページの「面」への分解とデータ分類

本番ではページを**信頼境界(trust boundary)で3面に再編**する：

## 3. 各論

### 3.1 顔認証（生体）の扱い — 案A（非保有）
改正APPIで顔認証データ・顔特徴量は**個人識別符号**＝それ自体が個人情報で機微性が高い。`insights.js` が挙げる Butlr/Density 型（サーマル/mmWaveのカメラレス占有計測）も、**ゾーン別占有は非生体センサで取る**方が筋が良い。生体DBを持たない＝漏洩時に「顔は変えられない」という回復不能リスクを構造的に排除する。member.js の「来館データの会員還元」（ランク/連続/割引）は会員ID＋時刻だけで全実装可能なので、生体を持つ動機が無い。

### 3.2 秘密管理 — 現状の止血と本番化
- **即時**: Maps鍵に Google Cloud で「HTTPリファラ制限（本番ドメイン＋localhost）」「Maps JavaScript APIのみ」「日次クォータ上限」。これだけで流出時の悪用（課金爆発・他API悪用）をほぼ封じる。
- **本番**: 理想は Maps 呼び出しをバックエンドプロキシ経由にして鍵をクライアントに出さない。ただし Maps JavaScript API はブラウザ実行が前提でプロキシ化が難しいため、**現実解はリファラ＋API＋クォータ制限で十分**（鍵自体は公開前提の設計）。それ以外のサーバ側秘密（HMACソルト、hacomono APIキー、Supabaseサービスロールキー）は **Secrets Manager / 環境変数**に置き、クライアントには絶対に出さない。
- **ローテ手順**を1枚に常備（漏洩検知→新鍵発行→env更新→旧鍵失効）。

### 3.3 認証・認可
- auth.js の擬似ログイン → **Supabase Auth の SMS OTP / パスキー**へ。電話番号の暗号化・レート制限・総当たり対策を自前で書かない。
- 認可は **Postgres Row Level Security** で「会員は自分の行だけ」「B2B契約先は集計ビューだけ」をDB層で強制。アプリ層のバグでも越境させない。
- hacomono とは OIDC/SSO 連携キーで紐付け、決済・本人確認PIIは hacomono に残す（保管点を増やさない）。

### 3.4 同意（consent.js を活かす）
現状の `consent.js` は**良いオプトイン設計**（既定オフ・明示的肯定操作でのみオン＝affirmative consent、コメント49行に明記）。本番化で足すのは：(1) 同意を **2層に分離**（「①サービス提供のための利用」と「②分析二次利用・第三者提供＝マチミル」）、(2) 同意状態を**サーバ側にも保存**し撤回を全ゾーンへ即時伝播、(3) **撤回ログ**を監査可能に保持。利用目的の特定（APPI第17条）と第三者提供の整理（同第27条/匿名加工なら第43-46条）をポリシーに明記。

### 3.5 B2B/マーケットプレイスの再識別防御（多層）
1. **k-匿名化**: dbtで `店舗×時間帯×属性` の最小群 **k>=10** 未満セルは**抑制(suppress)**。年齢は帯へ一般化。
2. **差分プライバシー**: カウントに Laplace ノイズ（`stats.js:129 dpCount` を本番移植）、**εバジェット**を集計単位で管理。
3. **データクリーンルーム**: BigQuery Analytics Hub等で**集計クエリのみ**許可、生レコードは外に出さない（ファイル受け渡しより安全）。
4. **契約・監査**: 提供契約に「再識別・名寄せ・他データ突合の禁止」「目的外利用禁止」「監査権」「漏洩時の連帯責任」。匿名加工情報として提供するなら作成基準・公表・識別行為禁止を遵守。
5. **実分布で再評価**: 合成データのk値で安心しない。松山の小population実分布で定期的にk値・再識別リスクを再計算。

### 3.6 ログ・インシデント
アクセス監査ログ（誰がどのデータに触れたか）を Supabase/BigQuery に保持。漏洩時は **APPI第26条の個人情報保護委員会報告・本人通知フロー**を雛形化。地方小規模なので「完璧な24/7監視」ではなく「検知→止血→鍵ローテ→報告」の最小手順を属人化しないチェックリストにする。

## 4. 「最小十分」の線引き（地方小規模が現実に守れる範囲）
やる：PIIをSupabase1箇所に集約／生体は非保有／同意2層＋撤回伝播／B2Bは多層匿名化＋クリーンルーム＋契約／Maps鍵リファラ制限／半期チェックリスト運用。
やらない（過剰）：自前生体DB、自前KMS/HSM運用、24/7 SOC、フルISMS認証。これらは規模に対し運用負荷とコストが過大で、むしろ形骸化リスクを生む。**「集める場所を絞り、危険なもの（生体・第三者提供）だけ厳格に、残りは非PII射影」**が本件の最適点。
