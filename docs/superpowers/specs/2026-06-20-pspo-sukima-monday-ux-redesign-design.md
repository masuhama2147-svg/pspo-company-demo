status: approved (brainstorming) → implementation

## 0. ゴール（このリデザインで何を達成するか）
**「使いやすさ ＝ UX を最大化」を最上位の評価軸**に、現行 SUKIMA の骨格（ビルド無し・5層CSS・config単一真実源・ES Modules・非PII・PWA・a11y）を**壊さずに**、顧客アプリ（`index.html` / `area.html` / `live.html` / `me.html`）を次の5点で進化させる。

1. **カメラレス化** — 計測方法（カメラ/AI/顔認証）を顧客アプリから全削除。訴求を機能ベースへ。
2. **antigravity 級モーション（ハイブリッド）** — PSPO のネイビー/オレンジ・ブランドは維持しつつ、Aurora 背景・スクロール連動 reveal・スプリング系マイクロインタラクションで"格"を引き上げる。
3. **初回オンボ・ウィザード（人間工学）** — 用途・ホーム店舗・時間帯を選ぶと全画面がパーソナライズ。
4. **ワンタップ・ログイン ＋ PWA 完結** — hacomono の外部ブラウザ蹴り出しを解消する体験を提示。
5. **住所手入力 → 近い順（位置情報不使用）** — 内蔵ガゼッタ＋ハバーサインで距離(km)付き並び替え。

**管理者レイヤー（顧客非開示）:** 混雑＝顔認証入退館の純人数(in−out)を唯一の正とする裏側データ方針を `growth.html` / `critique.html` のデータ章に明記（顧客アプリには一切出さない）。

### 非ゴール（YAGNI）
- 実バックエンド/実センサ/本物の認証は作らない（すべてデモの擬似）。
- 経営AI/批評ページの全面リデザインはしない（データ章の文言更新のみ）。
- ネイティブアプリ化はしない（PWA で「アプリ感」を出す）。

## 1. UX 原則（全実装が従う設計憲法）
ユーザー要望「UX 最大化」を、検証可能な原則に翻訳する。各実装はこれで自己採点する。

| 原則 | 適用 |
|---|---|
| **Fitts の法則** | 主要タップ標的は最小 56px、親指届く位置に。CTA は画面下部固定。 |
| **Hick の法則** | 選択肢は一度に最大5。オンボの用途選択は5枠に厳選。 |
| **親指ゾーン** | 片手操作前提。一次アクションは下 1/3、破壊的操作は上部/離れた位置。 |
| **段階的開示** | オンボは1問1画面。詳細（計測の仕組み等）は顧客には出さない。 |
| **Doherty の閾値** | 反応は 400ms 以内。タップに即座の視覚フィードバック（スプリング/トースト）。 |
| **Jakob の法則** | 地図・ボトムシート・タブバー・検索は既知の定番UIを踏襲（学習コストゼロ）。 |
| **Tesler の法則** | 複雑さは消えない＝アプリ側が引き受ける（座標解決・近い順計算は裏で）。 |
| **Miller の法則** | 1画面の情報塊は 7±2。カードで分節化。 |
| **ピーク・エンドの法則** | オンボ完了・チェックイン成功に"ご褒美"の山場（紙吹雪/+pt/達成）。 |
| **Aesthetic-Usability** | 美しさ自体が使いやすさの知覚を上げる＝モーションの"格"は機能。 |
| **アクセシビリティ** | reduced-motion 完全尊重・44px↑・16px入力・aria・skip-link・コントラスト維持。 |

## 2. デザインシステム — ハイブリッド「Aurora × ネイビー/オレンジ」
変更ファイル: `css/base.css`（トークン追加）, `css/motion.css`（大幅増強）, `css/components.css`（hover/depth調整）。配色トークン（`--accent` ネイビー / `--hot` オレンジ / `--accent-2` ティール）は**変更しない**。

### 2.1 Aurora 背景（CSSのみ・WebGL不要）
- `body` の既存 radial-gradient を、ネイビー基調に teal↔orange の柔らかな光が**ゆっくり流れる** aurora へ発展。`@keyframes auraFloat` を拡張し、複数レイヤーを別速度で動かす。
- ヒーローに `--aura-1/2` の blob（既存 `hero-brand__blob` を活用）。`prefers-reduced-motion` で停止。

### 2.2 スクロール連動 reveal / 視差
- 既存 `.io`（IntersectionObserver）を維持しつつ、対応ブラウザでは **CSS `animation-timeline: view()`** によるスクロール連動の fade/slide/scale へ格上げ（JS不要で滑らか）。非対応は現行の `.in` フォールバック。
- セクション見出し（大型 Anton 英字）は入場時に控えめな視差。

### 2.3 マイクロインタラクション
- ボタン: スプリング系 `transform` hover/active（`cubic-bezier(.2,.8,.2,1)`）＋押下の即時凹み（Doherty）。
- カード: hover で僅かな lift＋光沢。`prefers-reduced-motion` で無効化。
- ページ遷移: 対応ブラウザで `@view-transition { navigation: auto }` を有効化（顧客アプリ内の遷移を滑らかに）。

### 2.4 大型タイポ
- ヒーローの英字（`NEVER STOP` 等 Anton）を主役化。日本語見出しは Zen Kaku Gothic New 維持。

## 4. 初回オンボ・ウィザード（人間工学）＝ 新規 `js/onboard.js`
初回起動でフルスクリーン・ウィザードを表示（`localStorage: sukima-onboarded`）。スキップ可。完了で `sukima-profile`（既存キー）＋新規 `sukima-prefs` に保存し、ホーム/レコメンド/目標/me を即パーソナライズ。

### フロー（1問1画面＝段階的開示）
- **Step 1 — 何を目的に通う？**（複数可・Hickで5枠）: 🏋️ 筋トレ / 📚 自習・作業 / 🧖 サウナで整う / ☕ カフェ・居場所 / 🌙 ながら習慣化。
- **Step 2 — よく行く店舗**: 住所/地名を1入力 → 近い順（§5の geo を再利用）→ 1タップ選択。スキップで「松山大街道店」既定。
- **Step 3 — 通う時間帯**: 🌅 朝 / ☀️ 昼 / 🌙 夜（3択）。
- **完了**: ピーク・エンド演出（達成アニメ＋「準備OK」）→ ホームへ。

### 人間工学の実装
- 進捗ドット（3）で見通し。各画面 CTA は**最下部固定・56px**。戻る/スキップ常設。タップに即時フィードバック。
- 選択は即ハイライト（Doherty）。`role="dialog"` ＋ focus trap ＋ Esc/スキップ。reduced-motion 尊重。

### パーソナライズの反映
- 用途 → ホームの優先カード並び・me の目標 kind・レコメンド文面。
- 既存の `PERSONAS`/`member.js` ロジックに、選択 prefs から最も近いペルソナを**自動マッピング**（固定ペルソナを"自分で選ぶ"へ進化）。後から me で変更可。

## 5. 住所手入力 → 近い順（位置情報不使用）＝ 新規 `js/geo.js`
### 5.1 ガゼッタ（内蔵・Geocoding API 不要）
- 愛媛の市区町村・主要地名・主要駅 → 代表座標(lat/lng) の静的テーブル（松山/松山市駅/大街道/道後/空港通り/東温/砥部/伊予/松前/西条/新居浜/今治/しまなみ/宇和島/大洲/内子… ＋ 愛知半田・北海道）。
- 入力テキストに地名/駅が含まれれば即解決（部分一致・かな/漢字許容）。該当なしは「松山中心」へフォールバック＋注記。

### 5.2 距離
- **ハバーサイン**で各店舗との距離(km)。近い順ソート。各行に「◯◯km」表示。

### 5.3 前提修正 — `stores-real.js` の座標補正（重要）
- 現状、多くの店舗が同一ダミー点（例 `33.839519,132.765352`）に固まり、距離ソートが機能しない。
- **対応**: 各店舗の住所から市区町村/エリアをパースし、§5.1 ガゼッタの**エリア重心＋住所ハッシュ由来の小オフセット**で概略座標を再生成（同エリアは近接・別エリアは分離）。主要旗艦店は精緻座標をハードコード。
- UI に「位置は概略（エリア単位）」を**正直に注記**。実運用では Geocoding API で精緻化、と spec に明記。

### 5.4 area.html UI
- 既存検索＋エリア/カテゴリ chips の上に **「現在地は使わず、住所・駅で探す」入力**を追加。入力で近い順に並び替え＋各行 km 表示。空入力時は現行のエリア別グルーピング表示を維持。
- 地図 FAB の `navigator.geolocation`（[js/widgets.js:100-107](js/widgets.js#L100)）は残すが、近い順機能は**位置情報に依存しない**ことを明示。

## 6. ワンタップ・ログイン ＋ PWA 完結
### 6.1 ログイン画面（新規 `login.html` or 全画面モーダル）
- 電話番号入力 →「ログイン」1タップ、または「パスキーで続ける」風ボタン（**擬似認証**・外部遷移ゼロ）。`localStorage: sukima-auth`。
- ログイン済みは自動スキップ。ログアウト導線は me に。
- 「顔認証」という語は使わない（§3）。チェックインは「チェックイン」。

### 6.2 PWA 完結の明示
- 顧客アプリ内の遷移は**すべて内部**（外部 `target=_blank` は公式サイト/経路のみ）。
- critique に hacomono の具体（マイページが別SaaS＝外部ブラウザで開くだけ）を明記し、SUKIMA＝1 PWA で完結を解として提示。

## 7. 管理者レイヤー（顧客非開示）— 裏側データ方針
`growth.html` / `critique.html` のデータ章を更新（顧客アプリには出さない）:
- 混雑＝顔認証入退館の**純人数(in−out)** を唯一の正に。多カメラの**重複カウントを回避**、低コストで正確、カメラ非依存。
- 「多数カメラ→データ有用性が低い」の裏返しの解として提示。会員には計測方法を非開示、と設計判断に明記。

## 10. 受け入れ基準（Done の定義）
1. 顧客面（index/area/live/me）に計測方法の語が**ゼロ**（§3 grep）。
2. area で住所/駅を入れると**近い順＋km** に並ぶ。同エリア店舗が上位に来る。
3. 初回起動でオンボが出て、用途/店舗/時間帯の選択が**ホーム・me に反映**される。2回目以降は出ない。
4. ログインが**1タップ・外部遷移ゼロ**で完了し、PWA としてホーム画面追加できる。
5. Aurora/モーションが効き、`prefers-reduced-motion: reduce` で**全停止**する。
6. モバイル 390px で主要 CTA が**親指ゾーン・56px↑**、横スクロール無し。
7. 全ページ **console error 0**（Playwright/手動）。
8. config.js に**生APIキーが無い**。
9. 管理者ページに in−out 純人数の方針が記載され、顧客面には**出ていない**。
